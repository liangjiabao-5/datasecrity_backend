# 后端接口文档

## 1. 接口约定

### 1.1 基础路径

```text
/api/v1
```

### 1.2 认证

所有业务接口默认需要登录态。

请求头：

```http
Authorization: Bearer <accessToken>
```

### 1.3 统一响应

```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": {},
  "traceId": "f6a7a1f2b6"
}
```

错误示例：

```json
{
  "code": "PROJECT_CODE_DUPLICATED",
  "message": "项目编号已存在",
  "data": null,
  "traceId": "f6a7a1f2b6"
}
```

### 1.4 分页响应

```json
{
  "list": [],
  "pageNo": 1,
  "pageSize": 10,
  "total": 0
}
```

### 1.5 枚举建议

| 枚举 | 值 |
| --- | --- |
| 项目状态 | `NOT_STARTED`、`IN_PROGRESS`、`COMPLETED`、`ARCHIVED` |
| 评估结果 | `COMPLIANT`、`PARTIAL`、`NON_COMPLIANT`、`NOT_APPLICABLE` |
| 报告状态 | `PENDING`、`RUNNING`、`SUCCESS`、`FAILED`、`DELETED` |
| 知识库状态 | `ENABLED`、`DISABLED`、`DRAFT` |
| 风险等级 | `MAJOR`、`HIGH`、`MEDIUM`、`LOW`、`SLIGHT` |
| 数据处理活动 | `COLLECT`、`TRANSFER`、`STORE`、`USE_PROCESS`、`PROVIDE`、`PUBLIC`、`DELETE` |

### 1.6 本批实现状态

一期后端当前优先完成项目流程联调路径：

- 已实现：项目管理、项目基本信息、评估方案、信息调研、现场测评、汇总分析。
- 已实现：开始评估时生成项目检查项快照，重复开始评估不会重复生成。
- 已实现：现场测评 Excel 模板导出、Excel 导入、结果导出；报告管理 Word 模板生成、就绪检查、后台任务、状态轮询、失败重试、下载和删除；文件本地/MinIO 保存与下载。
- 暂缓实现：项目配置快照、项目成员、项目模板保存。
- 占位实现：大模型辅助接口暂不调用外部模型供应商；基础信息、评估方案、信息调研的正式文件导入导出仍由前端 JSON 本地兜底。

## 2. 项目管理接口

### 2.1 查询项目列表

```http
GET /api/v1/projects
```

参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| pageNo | number | 是 | 页码 |
| pageSize | number | 是 | 每页条数 |
| status | string | 否 | 项目状态 |
| keyword | string | 否 | 项目名称、编号、评估单位 |

返回：

```json
{
  "list": [
    {
      "id": "p001",
      "projectName": "XX银行数据安全评估项目",
      "projectCode": "ENST-2024-001",
      "assessmentOrg": "XX银行股份有限公司",
      "status": "IN_PROGRESS",
      "statusName": "进行中"
    }
  ],
  "pageNo": 1,
  "pageSize": 10,
  "total": 24
}
```

### 2.2 查询项目统计

```http
GET /api/v1/projects/statistics
```

返回：

```json
{
  "all": 24,
  "notStarted": 12,
  "inProgress": 8,
  "completed": 4
}
```

### 2.3 创建项目

```http
POST /api/v1/projects
```

请求：

```json
{
  "projectName": "XX银行数据安全评估项目",
  "projectCode": "ENST-2024-001",
  "assessmentOrg": "XX银行股份有限公司",
  "riskMatrixId": "matrix-v1",
  "assessmentTemplateId": "tpl-gb",
  "systemType": "MANAGEMENT_SYSTEM",
  "harmModelId": "harm-default",
  "scoreModelId": "score-v1",
  "description": "项目背景、目标和范围"
}
```

返回：

```json
{
  "projectId": "p001"
}
```

### 2.4 编辑项目

```http
PUT /api/v1/projects/{projectId}
```

说明：字段同创建项目。若修改评估模板、评分模型、风险危害程度模型或风险评价矩阵，后端应返回影响提示或在保存前进行校验。

### 2.5 查询项目详情

```http
GET /api/v1/projects/{projectId}
```

返回项目基础信息和项目当前配置字段。本批暂不返回项目配置快照、项目成员和项目模板信息。

### 2.6 开始评估

```http
POST /api/v1/projects/{projectId}/start
```

处理：

- 项目状态变更为 `IN_PROGRESS`。
- 如项目尚未生成检查项快照，则按评估模板生成项目检查项。
- 如项目已存在检查项快照，接口保持幂等，不重复生成。

### 2.7 删除项目

```http
DELETE /api/v1/projects/{projectId}
```

说明：逻辑删除。已生成报告的项目需保留报告审计记录。

### 2.8 保存为项目模板（后续批次）

```http
POST /api/v1/projects/{projectId}/save-template
```

说明：本批暂缓实现项目模板保存能力，后续批次再开放。

请求：

```json
{
  "templateName": "电力行业评估项目模板",
  "templateDescription": "适用于电力行业数据安全评估",
  "includeAssessmentTemplate": true,
  "includeScoreModel": true,
  "includeTeamMembers": false
}
```

### 6.5 分数计算

```http
POST /api/v1/projects/{projectId}/evaluation/calculate-score
```

计算口径：

- 读取项目绑定的 `scoreModelId` 对应评分模型。
- `score = (符合项数量 * COMPLIANT 分值 + 基本符合项数量 * PARTIAL 分值 + 不符合项数量 * NON_COMPLIANT 分值) / (符合项数量 + 基本符合项数量 + 不符合项数量) * 100`。
- `NOT_APPLICABLE` 不计入分子和分母。
- 使用评分模型 `possibilityRanges` 映射 `possibilityLevel`，区间按 `min <= score < max` 判断；为覆盖满分，`score = 100` 且区间 `max = 100` 时命中该区间。

返回：

```json
{
  "score": 85,
  "possibilityLevel": "LOW",
  "possibilityLevelName": "低",
  "scoreModelVersion": 3,
  "detail": {
    "compliantCount": 80,
    "partialCount": 20,
    "nonCompliantCount": 10,
    "notApplicableCount": 5
  }
}
```

## 8. 报告管理接口

说明：报告管理已按 `doc/数据安全风险评估报告模版.docx` 生成正式 Word 报告。模板批注标注的数据来源会从项目基本信息、评估方案、信息调研、现场测评、汇总分析及项目绑定的评分模型、风险评价矩阵中读取。

生成规则：

- 默认生成完整报告，也可通过 `selectedSections` 选择章节块。
- 动态表格严格沿用模板表头和列数，通过克隆模板数据行保留格式。
- 5.2“评估对象和范围”使用 `processing_activity_survey.involved_activities` 生成处理阶段，并将英文枚举转换为中文；下方表格固定保留模板唯一数据行，仅回填项目基本信息中的 `projectName` 和业务系统 `businessFunction`，序号和涉及数据类型保留模板原始内容。
- 7.2.4“数据处理活动情况”按 `processing_activity_survey.involved_activities` 生成中文阶段总述，并按已勾选活动动态生成逐阶段详情。
- 8.3.3.1“XX系统的综合得分及风险发生可能等级”标题中的系统名称使用项目基本信息中的 `projectName`。
- 报告中的“尚未填写”“未填写”“尚未配置”“未上传”和缺省 `-` 使用红色字体标识。
- 拓扑图和数据流转图等比缩放，最大宽度 6 英寸、最大高度 4.5 英寸。
- 模板样式、页眉页脚和固定内容不重建；生成完成后删除全部 Word 批注部件和批注标记。
- 可选数据表为空时删除对应表格和标题，并统一重排表格编号和文内引用；生成文件打开时自动刷新目录和域引用。
- 报告任务状态为 `PENDING`、`RUNNING`、`SUCCESS`、`FAILED`；报告逻辑删除前写入 `DELETED`。
- 同一项目可多次生成并保留多份报告记录。失败报告可重试，重试会新建任务并复用原报告记录。
- 生成报告、生成成功/失败、重试、下载、删除均写入审计日志。

前端推荐流程：

1. 调用就绪检查接口，展示 `warnings` 和阻断项。
2. 调用生成接口，取得 `reportTaskId` 和 `taskStatusUrl`。
3. 每 1-2 秒轮询任务状态，直到 `SUCCESS` 或 `FAILED`。
4. 成功后展示下载操作；失败后展示 `errorMessage` 和重试操作。
5. 页面刷新或重新进入时，通过报告列表恢复全部任务结果。

评分模型保存请求：

```json
{
  "modelName": "V1评分模型",
  "modelType": "PRESET",
  "resultScores": {
    "COMPLIANT": 1,
    "PARTIAL": 0.5,
    "NON_COMPLIANT": 0,
    "NOT_APPLICABLE": null
  },
  "possibilityRanges": [
    { "level": "HIGH", "min": 0, "max": 60, "includeMin": true, "includeMax": false },
    { "level": "MEDIUM", "min": 60, "max": 80, "includeMin": true, "includeMax": false },
    { "level": "LOW", "min": 80, "max": 100, "includeMin": true, "includeMax": true }
  ]
}
```

列表和详情返回均包含 `resultScores` 与 `possibilityRanges`，用于前端“查看评分模型规则”弹窗展示。当前知识库页面只展示查看入口，后端仍保留新增、编辑、删除接口供后续维护使用。

校验规则：

- `possibilityRanges` 必须覆盖 0%-100%。
- 相邻区间不得重叠、不得出现空档。
- 区间命中按 `min <= score < max` 执行；为覆盖满分，`score = 100` 且区间 `max = 100` 时命中该区间。
- `includeMin`、`includeMax` 字段保留兼容返回，后端计算以 `min`、`max` 和上述规则为准。
- 保存或更新评分模型时自动执行校验，失败则不写入。

  "remark": "默认 3x5 风险评价矩阵",

      "VERY_HIGH": "MAJOR",

      "MEDIUM": "LOW",
      "LOW": "SLIGHT"
    },
    "LOW": {
      "VERY_HIGH": "MEDIUM",

校验规则：

- 矩阵必须为非空二维对象。

- 单元格风险等级必须属于 `MAJOR`、`HIGH`、`MEDIUM`、`LOW`、`SLIGHT`。

## 10. 文件接口

说明：文件接口已支持本地文件对象保存和 MinIO 对象存储。信息调研拓扑图、数据流转图和报告文件对应的 `SURVEY_TOPOLOGY_DIAGRAM`、`SURVEY_DATA_FLOW_DIAGRAM`、`REPORT` 文件业务类型，在配置完整的 `MINIO_ENDPOINT`、`MINIO_ACCESS_KEY`、`MINIO_SECRET_KEY`、`MINIO_BUCKET_NAME` 后写入 MinIO；未配置 MinIO 时回退到后端 `instance/generated` 目录，也可通过 `FILE_STORAGE_ROOT` 配置覆盖。现场测评导入导出文件等其他业务类型仍保持本地存储。文件元数据写入 `file_object`。
