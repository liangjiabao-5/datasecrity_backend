# 项目编号允许重复功能修改说明

## 1. 背景

当前一个业务项目下允许录入多个系统项目，这些系统项目可能共用同一个项目编号。后端已按“小改法”调整：保留 `projectCode` 字段作为业务编号和展示字段，但不再要求全局唯一；项目唯一识别继续使用 `id` 或 `projectId`。

## 2. 后端调整

- `project.project_code` 由唯一索引调整为普通索引，允许重复。
- 创建项目时不再校验 `projectCode` 是否已存在。
- 编辑项目时允许把 `projectCode` 修改为已有编号。
- 项目列表、详情、基本信息、报告和导入导出中的项目编号字段名称不变。
- 项目子表、现场测评、风险汇总、报告等业务关联仍使用 `project_id`，不使用 `projectCode` 做关联。

## 3. 受影响接口

| 接口 | 调整点 |
| --- | --- |
| `POST /api/v1/projects` | 可创建与已有项目相同 `projectCode` 的新项目，返回新的 `projectId` |
| `PUT /api/v1/projects/{projectId}` | 可把当前项目的 `projectCode` 修改为已有编号 |
| `GET /api/v1/projects` | 按 `keyword` 搜索项目编号时，可能返回多条相同 `projectCode` 的项目 |
| `GET /api/v1/projects/{projectId}` | 路径参数仍为项目 ID，返回字段不变 |
| `GET /api/v1/projects/{projectId}/basic-info` | `projectNumber` 仍映射 `projectCode`，只读展示且允许重复 |
| `PUT /api/v1/projects/{projectId}/basic-info` | 保存基本信息仍以 `projectId` 定位项目 |

## 4. 前端调整要求

- 不要再对项目编号做唯一性校验；创建和编辑表单允许输入已有 `projectCode`。
- 项目列表行唯一键使用 `id`，不要使用 `projectCode`。
- 路由参数、接口调用参数、Tab Key、选中项 Key、缓存 Key、Map Key 均使用 `id` 或 `projectId`。
- 如果列表中出现相同项目编号，需要结合项目名称、系统类型、评估单位、状态、创建时间或项目 ID 做区分展示。
- 搜索项目编号时，需要允许返回多条记录，不要只保留第一条。
- 基本信息页的 `projectNumber` 仍作为展示字段处理，不要用它反查或定位项目。

## 5. 前端验证建议

1. 创建两个项目，填写相同 `projectCode`，确认均创建成功且返回不同 `projectId`。
2. 编辑其中一个项目，把项目编号改成另一个项目已有编号，确认保存成功。
3. 在项目列表按项目编号搜索，确认能展示多条相同编号的记录。
4. 分别进入两个同编号项目的现场测评、汇总分析、报告管理，确认数据互不串联。
5. 刷新页面或切换 Tab 后，确认前端仍按 `projectId` 恢复当前项目上下文。

## 6. 兼容说明

本次不修改接口字段名和响应结构。前端主要调整唯一性假设：`projectCode` 只代表业务编号，不代表系统唯一主键；后续所有业务定位以 `projectId/id` 为准。
