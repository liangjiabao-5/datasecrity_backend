# 现场测评符合情况多选查询后端接口修改说明

更新时间：2026-06-18

## 1. 背景

现场测评页面原来“符合情况”查询条件只支持单选，例如只能筛选 `不符合` 或 `基本符合` 其中一种。前端已调整为多选，用户可以同时选择多个符合情况，例如同时筛选 `不符合` 与 `基本符合` 的检查项。

本次不调整列表返回结构，只扩展查询参数。

## 2. 影响接口

```http
GET /api/v1/projects/{projectId}/evaluation/items
```

现有参数继续保留：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `pageNo` | number | 是 | 页码 |
| `pageSize` | number | 是 | 每页条数 |
| `keyword` | string | 否 | 检查编号、检查要点、检查内容关键词 |
| `sheet_name` | string | 否 | 一级目录/工作表名称筛选 |
| `sheetName` | string | 否 | `sheet_name` 的兼容字段 |
| `category` | string | 否 | 二级目录筛选 |
| `result` | string | 否 | 旧版单选符合情况筛选，建议继续兼容 |

新增参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `results` / `results[]` | string[] | 否 | 多选符合情况筛选 |

## 3. 枚举范围

`result` 和 `results` 中的值均使用现有 `evaluationResult` 枚举：

| 值 | 页面显示 |
| --- | --- |
| `COMPLIANT` | 符合 |
| `PARTIAL` | 基本符合 |
| `NON_COMPLIANT` | 不符合 |
| `NOT_APPLICABLE` | 不适用 |

## 4. 前端实际传参

前端使用 Axios 默认参数序列化。多选时会发送类似：

```http
GET /api/v1/projects/{projectId}/evaluation/items?pageNo=1&pageSize=10&results[]=PARTIAL&results[]=NON_COMPLIANT
```

后端建议同时兼容以下形式，降低框架绑定差异带来的联调成本：

```http
results[]=PARTIAL&results[]=NON_COMPLIANT
results=PARTIAL&results=NON_COMPLIANT
results=PARTIAL,NON_COMPLIANT
result=PARTIAL
```

其中 `result` 是旧版单选参数，建议保留兼容；当前新版前端优先发送 `results[]`。

## 5. 过滤规则

1. 未传 `results`、`results[]`、`result`，或数组为空时，不按符合情况过滤。
2. 传入一个或多个合法值时，按 `evaluation_result IN (...)` 过滤。
3. `results` 与其他查询条件之间为 AND 关系，例如同时受 `keyword`、`sheet_name`、`category` 约束。
4. 分页应在所有筛选条件生效后执行。
5. 如果同时传入 `results` 和 `result`，优先使用非空的 `results`；仅当 `results` 为空时再使用 `result`。
6. 建议对重复值去重。

示例 SQL 条件：

```sql
WHERE project_id = :projectId
  AND (:keyword IS NULL OR code LIKE :keyword OR check_point LIKE :keyword OR check_content LIKE :keyword)
  AND (:sheetName IS NULL OR sheet_name = :sheetName)
  AND (:category IS NULL OR category = :category)
  AND (:resultsEmpty = TRUE OR evaluation_result IN (:results))
ORDER BY sort_order ASC
LIMIT :pageSize OFFSET :offset;
```

## 6. 参数解析建议

后端如果是 Flask/FastAPI 一类框架，可以按下面思路归一化：

```python
raw_results = []
raw_results.extend(request.args.getlist("results[]"))
raw_results.extend(request.args.getlist("results"))

if not raw_results:
    comma_value = request.args.get("results")
    if comma_value:
        raw_results.extend(comma_value.split(","))

if not raw_results:
    single_result = request.args.get("result")
    if single_result:
        raw_results.append(single_result)

results = list(dict.fromkeys(value.strip() for value in raw_results if value and value.strip()))
```

后端如果是 Spring MVC，可以同时接收 `results[]` 与 `results`，再合并：

```java
@RequestParam(value = "results[]", required = false) List<String> bracketResults,
@RequestParam(value = "results", required = false) List<String> results,
@RequestParam(value = "result", required = false) String result
```

最终统一转成 `List<EvaluationResult>` 后参与查询。

## 7. 校验与错误处理

建议后端对 `result` / `results` 做枚举校验。

如果存在非法值，返回统一错误响应，例如：

```json
{
  "code": "INVALID_EVALUATION_RESULT_FILTER",
  "message": "符合情况筛选值只能为 COMPLIANT、PARTIAL、NON_COMPLIANT、NOT_APPLICABLE",
  "data": null
}
```

不建议静默忽略非法值，否则用户会误以为筛选条件已生效。

## 8. 返回结构

返回结构不变：

```json
{
  "list": [
    {
      "id": "item-001",
      "code": "1.1.1",
      "sheet_name": "数据安全管理",
      "category": "安全管理制度",
      "subcategory": "数据安全制度体系",
      "checkPoint": "检查要点",
      "checkContent": "检查内容",
      "evaluationResult": "NON_COMPLIANT",
      "evaluationRecord": "现场记录"
    }
  ],
  "pageNo": 1,
  "pageSize": 10,
  "total": 1
}
```

## 9. 验收用例

1. 不传 `results` 时，列表按原逻辑返回全部符合情况。
2. `results[]=NON_COMPLIANT` 时，仅返回“不符合”检查项。
3. `results[]=PARTIAL&results[]=NON_COMPLIANT` 时，同时返回“基本符合”和“不符合”检查项。
4. `results[]=PARTIAL&results[]=NON_COMPLIANT&keyword=制度` 时，仅返回符合情况命中且关键词命中的检查项。
5. `result=PARTIAL` 旧参数仍可查询“基本符合”检查项。
6. 同时传 `results[]=NON_COMPLIANT&result=PARTIAL` 时，以 `results[]` 为准，仅返回“不符合”检查项。
7. 传入非法值 `results[]=UNKNOWN` 时，返回参数校验错误。
