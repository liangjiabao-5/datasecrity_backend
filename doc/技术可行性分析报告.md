# 数据安全风险评估报告编制工具技术可行性分析报告

## 1. 项目概述

本项目建设目标是将数据安全风险评估过程中的项目管理、评估方案、信息调研、现场测评、汇总分析、报告生成和知识库管理串成一条可执行、可追溯、可复用的系统流程。当前需求说明和 HTML 原型已经明确了核心页面、标签页、按钮操作、字段范围和报告生成要求，具备进入技术设计和研发拆解的条件。

系统不是单纯的表单管理工具，关键价值在于三件事：

- 把评估过程数据结构化保存，避免报告编写时反复复制 Excel 和 Word。
- 把评分模型、风险危害程度分析模型、风险评价矩阵、处置建议和案例沉淀为知识库。
- 按项目最终确认数据自动生成数据安全风险评估报告，并保留报告生成记录。

## 2. 建设范围

本期按原型落地以下模块：

| 模块 | 页面来源 | 核心能力 |
| --- | --- | --- |
| 项目管理 | `项目管理页面.html`、`创建项目页面.html`、`编辑项目页面.html` | 项目创建、编辑、查看、状态管理、模板选择、模型选择、团队入口 |
| 项目基本信息 | `项目基本信息.html` | 项目基础资料、评估依据、参考标准、评估对象、被评估单位和联系人 |
| 评估方案 | `评估方案-评估团队.html` | 评估团队、被评估方团队、关注点、差距项 |
| 信息调研 | `信息调研.html` | 业务系统、数据资产、个人信息、重要/核心数据、处理活动、防护措施 |
| 现场测评 | `现场测评.html` | 检查项、评估结果、符合情况、分数计算、案例建议 |

系统设置仅在导航中出现，原型未给出业务内容，本期不作为详细开发范围。

## 3. 总体技术可行性结论

项目技术上可行，建议按“管理后台 + 业务后端 + 报告生成服务 + 可选大模型辅助服务”的方式建设。

推荐架构：

- 前端：Vue 3 + TypeScript + Vite + Element Plus。
- 后端：Python 3.11+ + Flask + SQLAlchemy。
- 数据库：MySQL 8 或 PostgreSQL 15，优先选择团队已有运维经验的数据库。
- 缓存与任务：Redis，用于异步任务状态、导入导出临时状态、报告生成任务锁。
- 文件存储：MinIO 或兼容 S3 的对象存储，用于拓扑图、流程图、导入导出文件、报告文件。
- 报告生成：后端任务服务生成 Word，模板填充优先采用 docx 模板引擎；复杂排版可接入 LibreOffice/OnlyOffice 做格式转换或预览。
- 大模型能力：作为可选辅助能力接入，用于案例匹配、评估记录辅助、整改建议生成、报告段落润色。所有大模型输出必须由用户确认后写入正式数据。

## 4. 技术方案选型

### 4.1 前端方案

| 方案 | 技术栈 | 优点 | 风险/成本 | 适用判断 |
| --- | --- | --- | --- | --- |
| 方案 A：Vue 管理后台 | Vue 3、TypeScript、Vite、Element Plus、Pinia | 中文后台生态成熟，表单、表格、弹窗、标签页实现效率高；团队容易上手 | 复杂表单需要统一封装，否则页面维护成本会上升 | 推荐方案 |
| 方案 B：React 管理后台 | React、TypeScript、Vite、Ant Design、Zustand | 组件能力强，工程化成熟 | 若团队 Vue 经验更多，会增加学习和协作成本 | 团队 React 经验充分时可选 |
| 方案 C：低代码/表单引擎 | amis、低代码平台或内部表单引擎 | 表单和列表产出快 | 复杂风险计算、报告生成和权限联动不容易完全覆盖 | 适合做管理配置页，不建议作为全系统主方案 |

建议采用方案 A。原型是典型管理后台形态，包含大量标签页、表单、列表、弹窗、批量操作、分页和导入导出，Element Plus 能直接覆盖主要交互。

### 4.2 后端方案

| 方案 | 技术栈 | 优点 | 风险/成本 | 适用判断 |
| --- | --- | --- | --- | --- |
| 方案 A：Python Flask 单体分层架构 | Flask、SQLAlchemy、Alembic、Pydantic/Marshmallow、MySQL/PostgreSQL、Redis、MinIO、Celery/RQ | 实现轻量，接口开发快；Python 生态适合 Word 报告、Excel 导入导出和大模型接入；单体分层便于一期快速闭环 | Flask 本身较轻，需要团队统一工程规范、权限、事务、异常和任务封装 | 推荐方案 |
| 方案 B：Python FastAPI 架构 | FastAPI、SQLAlchemy、Pydantic、Celery、MySQL/PostgreSQL | 类型声明和 OpenAPI 生成能力强，异步接口友好 | 用户已指定 Flask，且团队若以 Flask 为标准则不宜引入第二套 Web 框架 | 作为备选，不作为本次方案 |
| 方案 C：Python Django 架构 | Django、Django REST Framework、Django ORM、Celery、MySQL/PostgreSQL | ORM、后台管理和权限生态成熟，适合数据管理型系统 | 用户已指定 Flask；Django 框架约束更强，若只需轻量 REST 服务会显得偏重 | 可作为 Python 备选，不作为本次方案 |

建议采用方案 A。先用 Flask 单体分层架构把业务闭环跑通，通过 Blueprint 做领域模块拆分，通过 Celery 或 RQ 承载报告生成、导入导出、大模型调用等异步任务，保留文件服务和大模型服务的独立扩展接口。

### 4.3 报告生成方案

| 方案 | 实现方式 | 优点 | 风险/成本 | 建议 |
| --- | --- | --- | --- | --- |
| docx 模板填充 | 维护 Word 模板，后端按占位符、表格循环和条件块填充 | 与最终交付格式一致，便于业务人员维护模板 | 模板规则需要规范，复杂编号要统一处理 | 推荐 |
| HTML 转 Word/PDF | 先生成 HTML 报告，再转 Word/PDF | 前端展示和报告样式统一 | Word 格式可控性较差，编号和分页不稳定 | 可作为预览方案 |
| 人工导出 Excel + Word 手工编写 | 系统只提供数据导出 | 开发成本低 | 无法解决核心痛点 | 不建议 |

推荐采用 docx 模板填充。报告模板应定义章节块、表格块、编号块和可选内容块，系统生成报告时按项目最终确认数据填充。

### 4.4 大模型辅助方案

| 方案 | 内容 | 优点 | 风险/成本 | 建议 |
| --- | --- | --- | --- | --- |
| 不接入大模型 | 全部由规则和人工维护 | 实现简单，风险最低 | 案例建议和整改建议效率提升有限 | 可作为 MVP 保底 |
| 接入通用大模型 API | 用于建议生成、文本润色、案例相似度解释 | 开发快，效果好 | 需要脱敏、审计、输出可控 | 推荐作为增强能力 |
| 私有化模型/RAG | 企业内网模型 + 知识库检索 | 数据可控，适合敏感场景 | 建设成本高，模型效果和运维要求高 | 适合后续增强 |

建议一期预留大模型接口，可先实现可关闭的辅助能力。涉及项目敏感数据时，必须做脱敏、权限校验和人工确认。

## 5. 模块可行性分析

### 5.1 项目管理

可行。项目是全系统主数据，后端用项目主表承载项目编号、名称、评估单位、状态和模型配置，用项目配置快照保存模板、评分模型、风险危害程度模型和风险评价矩阵版本。前端按原型实现项目统计、状态筛选、列表、详情弹窗、创建、编辑和模板沉淀。

关键点：

- 项目编号必须唯一。
- 项目状态建议采用 `NOT_STARTED`、`IN_PROGRESS`、`COMPLETED`、`ARCHIVED`。
- 删除建议采用逻辑删除，避免报告和审计记录丢失。

### 5.2 项目基本信息与评估方案

可行。两部分以项目 ID 为主键关联，数据结构清晰。项目基本信息以单表或拆分表保存；评估团队、被评估方团队、关注点、差距项按子表保存。

关键点：

- 法律法规、标准规范既可引用知识库，也可保存项目自定义文本。
- 团队成员与系统用户不是同一概念，应允许录入外部人员。
- 关注点和差距项要记录来源，区分手工新增、案例选择和导入。

### 5.3 信息调研

可行，但字段多、表单深，是一期开发工作量较大的模块。建议把七类数据处理活动抽象为同一类对象，使用活动类型区分收集、传输、存储、使用和加工、提供、公开、删除。

关键点：

- 业务系统、数据资产、个人信息、重要数据、核心数据之间要有清晰关联。
- 拓扑图、数据流转图走统一文件上传接口。
- 导入数据需要返回逐行错误，不能只给一个失败提示。

### 5.4 现场测评

可行，是业务核心模块。检查项由项目选择的评估模板生成或快照化到项目中。现场测评记录按检查项保存“评估结果”（`evaluationRecord`）和“符合情况”（`evaluationResult`）。分数计算按项目评分模型执行。

关键点：

- 检查项模板变更不能影响已创建项目，应使用项目检查项快照。
- 批量修改要记录审计日志。
- 分数计算结果要记录使用的评分模型版本。

### 5.5 汇总分析

可行，但需要明确“自动生成”和“人工确认”的边界。系统从基本符合和不符合检查项生成一对一风险汇总记录，并根据国标风险源模板补全风险描述、整改建议、风险源描述和风险类型，再投影为风险源清单、风险清单和风险处置建议。风险等级默认留空，最终允许人工复核。

关键点：

- 刷新统计数据不能静默覆盖人工修改内容。
- 风险记录要能追溯到检查项和调研资产。
- 报告生成使用“最终确认版”风险清单。

### 5.6 报告管理

可行。建议采用异步任务。用户点击生成后创建报告任务，任务服务读取项目最终数据，填充 Word 模板，生成文件后写入报告记录。

关键点：

- 报告状态建议包含 `PENDING`、`RUNNING`、`SUCCESS`、`FAILED`。
- 失败时保存错误摘要，便于排查。
- 下载接口必须校验项目权限。

### 5.7 知识库管理

可行。知识库包括模板、模型、矩阵、建议、案例等配置。这里需要版本管理，否则项目使用中的模板和模型会被后续修改影响。

关键点：

- 被项目引用的知识库条目不建议物理删除，可停用。
- 评分模型区间必须覆盖 0%-100%，无重叠、无空档。
- 风险评价矩阵要保存可计算结构，而不是只保存展示文本。

## 6. 核心技术难点与处理策略

| 难点 | 风险 | 处理策略 |
| --- | --- | --- |
| 模板和模型版本变化 | 历史项目计算结果被新规则影响 | 项目创建时保存模板/模型/矩阵快照，计算结果绑定版本 |
| 信息调研表单复杂 | 前端页面难维护，导入失败难定位 | 统一表单组件和明细表组件，导入返回逐行错误 |
| 风险清单自动生成 | 自动生成可能覆盖人工修正 | 生成草稿与最终确认分离，刷新时提示覆盖范围 |
| Word 报告编号 | 章节、表格、图示编号容易错 | 由报告服务统一生成编号，不允许用户手工维护编号 |
| 大模型输出不可控 | 建议内容不准确或泄露敏感信息 | 仅作为辅助建议，输出需人工确认；请求前脱敏，记录审计 |
| 导入导出文件安全 | 文件格式错误、恶意文件、越权下载 | 文件类型校验、大小限制、病毒扫描预留、权限校验、对象存储隔离 |

## 7. 非功能可行性

### 7.1 性能

一期按管理后台系统设计，核心压力在列表查询、导入导出和报告生成。建议：

- 常规列表接口 1 秒内返回。
- 单项目检查项数量 1000 条以内时，现场测评分页查询稳定。
- 报告生成使用异步任务，不占用 HTTP 请求长连接。
- 导入导出大文件走任务化处理，前端轮询状态。

### 7.2 安全

- 所有接口必须鉴权。
- 项目数据按项目权限隔离。
- 文件下载必须校验项目访问权限。
- 大模型请求必须脱敏，不允许默认发送联系人、手机号、邮箱等个人信息。
- 删除、批量修改、重新计算、报告生成和报告删除记录审计日志。

### 7.3 可维护性

- 前端按页面模块拆分，公共表格、表单、弹窗、导入导出、模型选择器统一封装。
- 后端按领域模块分包，避免所有业务逻辑堆在一个 service。
- 报告模板和评分模型采用版本管理。
- 接口返回结构统一，便于前后端联调。

## 8. 成本与排期判断

建议按 3 个阶段建设：

| 阶段 | 周期建议 | 目标 |
| --- | --- | --- |
| MVP | 4-6 周 | 完成项目、基本信息、评估方案、知识库基础、现场测评基础、报告管理骨架 |
| 业务闭环 | 4-6 周 | 完成信息调研、分数计算、汇总分析、报告生成、导入导出 |
| 增强优化 | 3-4 周 | 完成案例建议、大模型辅助、复杂报告模板、审计、性能和权限细化 |

实际周期取决于报告模板复杂度、导入模板规范程度、历史数据是否需要迁移，以及大模型能力是否一期上线。

## 9. 结论

本项目技术可行，建议采用“Vue 3 管理前端 + Python Flask 单体分层后端 + MySQL/PostgreSQL + Redis + MinIO + Word 模板报告服务”的方案。一期重点是跑通项目评估和报告生成闭环，知识库和模型必须从一开始做版本管理。大模型能力可以作为可开关增强能力接入，但不能替代人工确认和规则计算。
