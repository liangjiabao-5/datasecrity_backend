# 风险源清单合并开关功能修改说明

更新时间：2026-07-16

## 一、调整范围

“数据安全风险源清单”页新增“是否合并”开关和“更新合并”按钮，由评估人员自行操作。

合并只影响以下两个页签的列表返回数据：

| 页签 | 接口 | 表结构 |
| --- | --- | --- |
| 数据安全风险清单 | `GET /api/v1/projects/{projectId}/risk-items` | 不变 |
| 数据安全风险处置建议 | `GET /api/v1/projects/{projectId}/risk-suggestions` | 不变 |

“数据安全风险源清单”页自身列表不合并，仍保持原明细数据。

## 二、新增接口

### 1. 查询合并开关状态

```http
GET /api/v1/projects/{projectId}/risk-summary/merge
```

响应：

```json
{
  "code": "SUCCESS",
  "data": {
    "mergeEnabled": false
  }
}
```

### 2. 设置合并开关状态

```http
PUT /api/v1/projects/{projectId}/risk-summary/merge
Content-Type: application/json

{
  "mergeEnabled": true
}
```

关闭合并时传 `false` 即可，关闭后风险清单和处置建议恢复原明细列表。

### 3. 更新合并数据

前端“更新合并”按钮调用：

```http
POST /api/v1/projects/{projectId}/risk-summary/merge/update
```

无冲突响应示例：

```json
{
  "code": "SUCCESS",
  "message": "更新合并数据成功",
  "data": {
    "mergeEnabled": true,
    "updatedMergedCount": 2,
    "hasMergeConflict": false,
    "mergeConflictTabs": []
  }
}
```

存在冲突响应示例：

```json
{
  "code": "SUCCESS",
  "message": "更新合并数据成功，存在合并冲突，数据安全风险清单标签页下的危害程度、发生可能性、风险等级字段，数据安全风险处置建议标签页下的整改建议字段存在冲突，请重新填写",
  "data": {
    "mergeEnabled": true,
    "updatedMergedCount": 2,
    "hasMergeConflict": true,
    "mergeConflictTabs": [
      {
        "tab": "数据安全风险清单",
        "fields": ["危害程度", "发生可能性", "风险等级"],
        "fieldCodes": ["harmLevel", "possibilityLevel", "riskLevel"]
      },
      {
        "tab": "数据安全风险处置建议",
        "fields": ["整改建议"],
        "fieldCodes": ["remediationSuggestion"]
      }
    ]
  }
}
```

前端点击“更新合并”后，直接使用响应 `message` 作为提示文案。

## 三、开启或更新合并前校验

开启合并或点击“更新合并”时，后端都会检查当前“数据安全风险源清单”的所有行：

- `relatedData`（涉及的数据及类型、级别）不能为空，也不能是默认值 `-`。
- `relatedActivities`（涉及的数据处理活动）不能为空数组。

如果存在未填写完整的行，接口返回 `400`，前端直接展示 `message`：

```json
{
  "code": "RISK_MERGE_REQUIRED_FIELDS_MISSING",
  "message": "第1、2行需填写完整后再进行合并",
  "data": {
    "rowNos": "1、2"
  }
}
```

## 四、合并规则

后端按以下字段完全一致进行合并：

| 合并判断字段 | 后端字段 |
| --- | --- |
| 评估类别 | `assessmentCategory` |
| 评估子类 | `assessmentSubcategory` |
| 风险类型 | `riskTypes` |
| 风险源类型 | `riskSourceType` |
| 涉及的数据处理活动 | `relatedActivities` |
| 涉及的数据及类型、级别 | `relatedData` |

合并后的列表表结构不变，仅以下字段会聚合：

| 字段 | 聚合方式 |
| --- | --- |
| 风险描述 `riskDescription` | 多条用换行符 `\n` 拼接去重 |
| 评估项ID `assessmentItemId` | 多条用 `、` 拼接去重 |
| 评估结果 `evaluationRecord` | 多条用换行符 `\n` 拼接去重 |
| 符合情况 `evaluationResult` | 固定返回 `NON_COMPLIANT` |

列表接口额外返回：

```json
{
  "merged": true,
  "mergedRiskRecordIds": ["riskrec-1", "riskrec-2"],
  "hasMergeConflict": false,
  "mergeConflictFields": []
}
```

这些字段供前端识别合并行使用，不需要作为表格列展示。

## 五、合并冲突说明

合并行背后的明细如果以下字段不一致，后端会标记冲突：

| 字段 | 后端字段 | 所属页签 |
| --- | --- | --- |
| 危害程度 | `harmLevel` | 数据安全风险清单 |
| 发生可能性 | `possibilityLevel` | 数据安全风险清单 |
| 风险等级 | `riskLevel` | 数据安全风险清单 |
| 整改建议 | `remediationSuggestion` | 数据安全风险处置建议 |

列表行冲突示例：

```json
{
  "merged": true,
  "mergedRiskRecordIds": ["riskrec-1", "riskrec-2"],
  "hasMergeConflict": true,
  "mergeConflictFields": ["harmLevel", "riskLevel", "remediationSuggestion"],
  "mergeConflictMessage": "合并行中存在不一致的评审结果，请评估人重新确认。"
}
```

“更新合并”按钮产生冲突时，提示文案必须使用接口响应 `message`，格式为：

```text
更新合并数据成功，存在合并冲突，xx标签页下的xx字段存在冲突，请重新填写
```

例如：

```text
更新合并数据成功，存在合并冲突，数据安全风险清单标签页下的危害程度、发生可能性、风险等级字段，数据安全风险处置建议标签页下的整改建议字段存在冲突，请重新填写
```

## 六、合并行修改与应用同步规则

合并开启后，“数据安全风险清单”和“数据安全风险处置建议”列表中返回的合并行会带上：

```json
{
  "merged": true,
  "mergedRiskRecordIds": ["riskrec-1", "riskrec-2"]
}
```

前端在合并行上执行以下操作时，会将 `mergedRiskRecordIds` 一并提交给后端：

| 页签 | 操作 | 接口 | 需要同步的字段 |
| --- | --- | --- | --- |
| 数据安全风险清单 | 修改 | `PUT /api/v1/projects/{projectId}/risk-items/{id}` | `harmLevel`、`possibilityLevel`、`riskLevel` |
| 数据安全风险清单 | 确认应用辅助判定 | `POST /api/v1/projects/{projectId}/risk-items/{id}/harm-analysis/apply` | `harmLevel`、`harmDescription`、`harmImpactObject`、`harmExample`、`harmAnalysisTrace`、`harmAnalysisConfidence`、`harmAnalysisInputHash`、`harmAnalysisStatus`、`riskLevel` |
| 数据安全风险处置建议 | 修改 | `PUT /api/v1/projects/{projectId}/risk-suggestions/{id}` | `remediationSuggestion` |

请求体示例：

```json
{
  "harmLevel": "HIGH",
  "possibilityLevel": "MEDIUM",
  "riskLevel": "HIGH",
  "mergedRiskRecordIds": ["riskrec-1", "riskrec-2"]
}
```

后端处理要求：

1. 如果请求体存在 `mergedRiskRecordIds` 且数组非空，后端必须以该数组作为更新目标，批量更新数组内所有原始风险记录。
2. 如果请求体不存在 `mergedRiskRecordIds` 或数组为空，后端保持原逻辑，仅更新路径参数 `{id}` 对应的记录。
3. `mergedRiskRecordIds` 仅作为定位合并源记录的传输字段，不应持久化覆盖到原始风险记录业务字段中。
4. 批量更新完成后，再次查询 `risk-items` 或 `risk-suggestions` 时，合并行应由这些原始记录重新聚合得到，避免合并行展示值与原始记录不一致。
5. 对合并行执行“确认应用辅助判定”时，后端应将确认后的危害程度及辅助判定详情同步到 `mergedRiskRecordIds` 内所有原始记录，并基于每条原始记录自身的发生可能性重新计算对应 `riskLevel`。
6. 对合并行执行“数据安全风险处置建议”修改时，后端应将 `remediationSuggestion` 同步到 `mergedRiskRecordIds` 内所有原始记录。

## 七、前端改造建议

1. “数据安全风险源清单”页新增“是否合并”开关和“更新合并”按钮。
2. 进入“数据安全风险源清单”页时调用 `GET /risk-summary/merge` 回显开关。
3. 用户打开或关闭开关时调用 `PUT /risk-summary/merge`。
4. 用户点击“更新合并”按钮时调用 `POST /risk-summary/merge/update`。
5. 如果接口返回 `RISK_MERGE_REQUIRED_FIELDS_MISSING`，保持开关为关闭状态，并直接展示接口 `message`。
6. 如果“更新合并”接口返回 `SUCCESS`，直接展示接口 `message`。
7. 如果 `data.hasMergeConflict` 为 `true`，可使用 `data.mergeConflictTabs[].fieldCodes` 高亮对应页签下的字段。
8. 开关设置成功或更新合并成功后，刷新“数据安全风险清单”和“数据安全风险处置建议”两个页签列表。
9. 在合并行上修改风险清单、应用辅助判定、修改处置建议时，请求体需要携带该行返回的 `mergedRiskRecordIds`，由后端同步更新所有原始风险记录。

## 八、验收点

1. 未开启合并时，风险清单和处置建议列表与原来一致。
2. 开启合并后，风险源清单仍显示原明细行。
3. 开启合并后，风险清单和处置建议按六个判断字段聚合。
4. 合并行的 `riskDescription`、`assessmentItemId`、`evaluationRecord` 已聚合，`evaluationResult` 为 `NON_COMPLIANT`。
5. 任一风险源行未填写 `relatedData` 或 `relatedActivities` 时，无法开启或更新合并，并以一句话提示所有缺失行号。
6. 点击“更新合并”后，无冲突时提示“更新合并数据成功”。
7. 点击“更新合并”后，有冲突时提示“更新合并数据成功，存在合并冲突，xx标签页下的xx字段存在冲突，请重新填写”。
8. 合并行修改风险清单或处置建议后，`mergedRiskRecordIds` 内所有原始风险记录的对应字段同步更新。
9. 合并行确认应用辅助判定后，`mergedRiskRecordIds` 内所有原始风险记录的危害程度、辅助判定详情和风险等级同步更新。
