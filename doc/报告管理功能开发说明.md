# 报告管理功能开发说明

## 1. 开发依据

报告管理模块依据以下文档和正式模板实现：

- `doc/后端技术开发指导及规范.md`
- `doc/技术可行性分析报告.md`
- `doc/数据安全报告编制需求说明.docx`
- `doc/研发任务拆解与实施计划.md`
- `doc/数据安全风险评估报告模版.docx`
- `原型/报告管理.html`

模块目标是完成“生成前检查 -> 创建报告任务 -> 状态轮询 -> Word 下载 -> 失败重试/逻辑删除”的完整闭环。

## 2. 已实现范围

- 独立报告领域蓝图 `app/blueprints/report.py`。
- 报告列表、生成前就绪检查、生成任务、任务状态、失败重试、下载和删除。
- 按正式 Word 模板批注标注的数据来源填充项目最终数据。
- 动态表格、可选表格、拓扑图、数据流转图、评分模型、风险矩阵、综合得分、风险清单和整改建议填充。
- 统一重排表格编号和文内表格引用。
- 生成后删除全部 Word 批注 XML、批注关系和正文批注标记。
- 报告任务成功、失败、重试、下载和删除审计。
- 报告文件优先写入 MinIO，未配置 MinIO 时回退到本地文件目录。

## 3. 前端页面交互

### 3.1 页面初始化

调用：

```http
GET /api/v1/projects/{projectId}/reports?pageNo=1&pageSize=10
```

列表主要展示字段：

| 页面列 | 接口字段 | 说明 |
| --- | --- | --- |
| 类型 | `reportTypeName` | 当前为“数据安全风险评估报告” |
| 生成状态 | `status` | `PENDING`、`RUNNING`、`SUCCESS`、`FAILED` |
| 生成时间 | `generatedAt` | 成功后有值 |
| 创建人 | `createdBy` | 发起生成的用户 |
| 失败原因 | `errorMessage` | 失败状态展示 |
| 操作 | `downloadUrl` / `reportId` | 成功下载、失败重试、非运行中删除 |

状态展示建议：

| 状态 | 中文 | 可用操作 |
| --- | --- | --- |
| `PENDING` | 待生成 | 刷新 |
| `RUNNING` | 生成中 | 刷新 |
| `SUCCESS` | 生成完毕 | 下载、删除 |
| `FAILED` | 生成失败 | 重试、删除 |

### 3.2 点击生成

1. 先调用 `GET /api/v1/projects/{projectId}/reports/readiness`。
2. `ready=false` 时展示 `blockers` 并禁止生成。
3. 存在 `warnings` 时弹出确认提示；用户确认后提交生成。
4. 调用 `POST /api/v1/projects/{projectId}/reports/generate`。
5. 保存返回的 `reportTaskId`，每 1-2 秒轮询任务接口。
6. `SUCCESS` 后停止轮询并刷新列表；`FAILED` 后展示 `errorMessage`。

生成请求：

```json
{
  "reportName": "XX公司数据安全风险评估报告",
  "reportType": "DATA_SECURITY_RISK_ASSESSMENT",
  "selectedSections": [
    "BASIC_INFO",
    "PLAN",
    "SURVEY",
    "EVALUATION",
    "RISK_SUMMARY",
    "SUGGESTIONS"
  ]
}
```

默认应全选全部章节。章节选择对应关系：

| 章节值 | 模板内容 |
| --- | --- |
| `BASIC_INFO` | 评估背景、目的、依据、框架、对象和范围 |
| `PLAN` | 评估人员、实施流程和评估方法 |
| `SURVEY` | 系统情况、数据资产、处理活动和安全措施 |
| `EVALUATION` | 风险识别和现场测评问题清单 |
| `RISK_SUMMARY` | 风险源、危害程度、可能性、评分、矩阵和风险清单 |
| `SUGGESTIONS` | 综合分析、风险处理和整改建议 |

### 3.3 失败重试

仅失败报告展示重试按钮：

```http
POST /api/v1/projects/{projectId}/reports/{reportId}/retry
```

重试返回新的 `reportTaskId`，前端继续使用任务状态接口轮询。

### 3.4 下载和删除

```http
GET /api/v1/projects/{projectId}/reports/{reportId}/download
DELETE /api/v1/projects/{projectId}/reports/{reportId}
```

- 仅 `SUCCESS` 报告可下载。
- `PENDING`、`RUNNING` 报告不可删除。
- 删除为逻辑删除，删除后列表不再展示且文件不可下载。

## 4. 模板填充规则

### 4.1 数据来源

| 报告内容 | 页面/标签页数据来源 |
| --- | --- |
| 项目编号、评估对象、系统描述、法规标准、时间计划 | 项目基本信息/项目基本情况 |
| 单位名称、地址、信用代码、邮编、负责人、联系人、简介 | 项目基本信息/被评估单位基本信息 |
| 评估团队、被评估方团队 | 评估方案对应标签页 |
| 系统情况、拓扑图、数据流转图 | 信息调研/业务和信息系统调研 |
| 数据资产、个人信息、重要数据、核心数据 | 信息调研/数据资产调研 |
| 数据处理活动 | 信息调研/数据处理活动调研 |
| 安全措施 | 信息调研/安全防护措施调研 |
| 指标分类、问题清单、综合得分 | 现场测评 |
| 风险源、危害程度、风险等级、整改建议 | 汇总分析最终数据 |
| 得分区间、结果分值、评价矩阵 | 项目绑定的评分模型和风险评价矩阵 |

章节专项填充规则：

- 5.2“评估对象和范围”正文中的数据处理阶段，来自“信息调研/数据处理活动调研”的 `activityTypes` 勾选结果。
- 5.2 下方业务系统表保留模板四列表头和列结构，数据行仅填充“系统名称”和“系统描述”；“系统描述”来自“项目基本信息/项目基本情况”。
- 7.2.4“数据处理活动情况”先使用 `activityTypes` 生成处理阶段总述，再按每个已勾选活动动态生成一条“XX阶段：处理详情。”；详情汇总该活动已填写的处理场景、方式、数据类型、数据级别、数据名称、数据规模、处理目的、处理频率和安全措施。

### 4.2 格式约束

- 所有动态表格严格使用模板已有表头，不增加或减少字段列。
- 5.2 业务系统表的数据行仅回填系统名称、系统描述，其余模板字段保持空白。
- 动态数据行由模板首条数据行克隆，沿用行高、边框、字体、段落和单元格样式。
- 个人信息、重要数据、核心数据表为空时，删除对应表格和表题。
- 图片保持原始宽高比，最大宽度 6 英寸、最大高度 4.5 英寸并居中。
- 不重建模板 `styles.xml`、页眉页脚、主题和固定图片。
- 报告生成完成后删除 `comments.xml`、扩展批注部件、关系声明和正文批注标记。
- 删除可选表格或未选章节后，统一重排表格编号和文内“表 N”引用。
- 生成文件设置 Word `updateFields=true`，打开报告时自动刷新目录和域引用。

## 5. 缺失数据处理

生成前就绪检查将问题分为：

- `blockers`：阻止生成，例如正式模板不存在。
- `warnings`：允许继续生成，例如团队、系统调研、现场测评或风险汇总为空。

存在警告但用户仍确认生成时，报告使用“尚未填写”“未填写”、`-` 或缺省数据行，不保留模板示例业务数据。所有代表尚未填写、尚未配置或未上传的回填内容均使用红色字体标识；仅修改对应文本的字体颜色，不改变模板原有段落、表格和样式文件。

## 6. 后端配置

```env
REPORT_TEMPLATE_PATH=doc/数据安全风险评估报告模版.docx
REPORT_TASK_ASYNC=true
```

- `REPORT_TASK_ASYNC=true`：接口创建任务后由后台执行器生成，前端轮询状态。
- `REPORT_TASK_ASYNC=false`：用于测试或本地调试，生成请求内执行，但接口结构保持不变。
- 当前工程使用进程内后台执行器完成一期闭环；多实例生产部署应替换为 Celery/RQ，任务入参继续只传 `projectId`、`reportId`、`taskId`。
