## 2. 后端需要新增的接口

### 2.1 数据安全风险源清单导出

```http
GET /api/v1/projects/{projectId}/risk-sources/export
```

返回：Excel 文件流。

建议文件名：`数据安全风险源清单-{projectId}.xlsx`

导出范围：当前项目下全部风险源清单数据，不只导出当前分页。

导出列：除页面“操作”列外，其余列表列均需包含，建议顺序如下：

| Excel 列名 | 后端字段建议 | 说明 |
| --- | --- | --- |
| 序号 | index | 导出时按当前排序生成；导入时可忽略 |
| 风险类型 | riskTypes | 多值可用顿号、逗号、分号或换行分隔 |
| 检查要点 | checkPoint | 来源于现场测评检查项 |
| 评估结果 | evaluationRecord | 来源于现场测评评估结果/记录 |
| 风险描述 | riskDescription | 可导入更新 |
| 风险源描述 | riskSourceDescription | 可导入更新 |
| 涉及的数据及类型、级别 | relatedData | 可导入更新 |
| 涉及的数据处理活动 | relatedActivities | 多值可用顿号、逗号、分号或换行分隔 |

如后端需要稳定定位被修改的风险源记录，建议在 Excel 中增加隐藏列 `riskSourceId` 或 `riskRecordId`；该列不作为页面展示列，前端不会依赖它展示。

### 2.2 数据安全风险源清单导入

```http
POST /api/v1/projects/{projectId}/risk-sources/import
Content-Type: multipart/form-data

file=@risk-sources.xlsx
```

导入要求：

- 支持 `.xlsx`，可兼容 `.xls`。
- 解析上述风险源导出列。
- “风险类型”和“涉及的数据处理活动”需要支持多值拆分。
- 建议仅允许更新风险类型、风险描述、风险源描述、涉及的数据及类型级别、涉及的数据处理活动等人工可调整字段。
- 检查要点、评估结果等来源字段建议用于定位或校验，不建议被导入覆盖。
- 导入成功后，前端会刷新风险源清单第一页。

返回结构：

```json
{
  "importedCount": 10,
  "failedCount": 0,
  "errors": []
}
```

出现部分失败时：

```json
{
  "importedCount": 8,
  "failedCount": 2,
  "errors": [
    {
      "rowNo": 3,
      "field": "风险类型",
      "reason": "风险类型值不在允许范围内"
    }
  ]
}
```

## 3. 数据安全风险清单接口

### 3.1 数据安全风险清单导出

```http
GET /api/v1/projects/{projectId}/risk-items/export
```

返回：Excel 文件流。

建议文件名：`数据安全风险清单-{projectId}.xlsx`

导出范围：当前项目下全部风险清单数据，不只导出当前分页。

导出列：除“判定状态、危害程度描述、影响对象、操作”四列外，其余列表列均需包含，建议顺序如下：

| Excel 列名 | 后端字段建议 | 说明 |
| --- | --- | --- |
| 序号 | index | 导出时按当前排序生成；导入时可忽略 |
| 风险类型 | riskTypes | 多值可用顿号、逗号、分号或换行分隔 |
| 风险描述 | riskDescription | 可导入更新 |
| 危害程度 | harmLevel | 支持“很高/高/较高/中/低/待评估”或枚举值 |
| 发生可能性 | possibilityLevel | 支持“高/中/低/待评估”或枚举值 |
| 风险源描述 | riskSourceDescription | 可导入更新 |
| 风险等级 | riskLevel | 导出最终风险等级；导入时建议后端按矩阵重算并返回最终值 |
| 涉及的数据及类型、级别 | relatedData | 可导入更新 |
| 涉及的数据处理活动 | relatedActivities | 多值可用顿号、逗号、分号或换行分隔 |

不导出的页面列：

- 判定状态：`harmAnalysisStatus`
- 危害程度描述：`harmDescription`
- 影响对象：`harmImpactObject`
- 操作列

如后端需要稳定定位被修改的风险记录，建议在 Excel 中增加隐藏列 `riskItemId` 或 `riskRecordId`；该列不作为页面展示列，前端不会依赖它展示。

### 3.2 数据安全风险清单导入

```http
POST /api/v1/projects/{projectId}/risk-items/import
Content-Type: multipart/form-data

file=@risk-items.xlsx
```

导入要求：

- 支持 `.xlsx`，可兼容 `.xls`。
- 解析上述风险清单导出列。
- 不解析、不更新判定状态、危害程度描述、影响对象。
- “风险类型”和“涉及的数据处理活动”需要支持多值拆分。
- “危害程度”和“发生可能性”导入后，应按项目风险矩阵重新计算风险等级；如果导入文件提供的风险等级与矩阵结果不一致，建议以后端计算结果为准。
- 导入成功后，前端会刷新风险清单第一页。

返回结构与风险源导入保持一致：

```json
{
  "importedCount": 10,
  "failedCount": 0,
  "errors": []
}
```

## 4. 前端联调行为

- 导出接口返回 Blob 文件流后，前端直接下载为 `.xlsx`。
- 导入接口使用 `multipart/form-data`，文件字段名固定为 `file`。
- 前端兼容接口直接返回导入结果，也兼容 `{ "data": { ... } }` 包装结构。
- `failedCount > 0` 或 `errors` 非空时，前端会弹出“汇总分析导入异常明细”，展示 `rowNo / field / reason`。
- 导入失败的 HTTP 响应建议返回统一错误结构中的 `message`，前端会直接展示。

## 5. 建议校验规则

- 空行应忽略。
- 风险类型、数据处理活动、危害程度、发生可能性、风险等级建议同时支持中文显示值和后端枚举值。
- 对无法定位原风险记录的行，应返回行级错误，不应新建孤立风险记录。
- 同一文件内重复定位到同一风险记录时，建议后出现的行返回重复错误。
- 导入过程建议保留部分成功模式：合法行写入，异常行进入 `errors`。
