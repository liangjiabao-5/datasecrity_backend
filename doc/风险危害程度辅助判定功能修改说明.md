# 风险危害程度辅助判定功能修改说明

更新时间：2026-06-02

## 一、已删除的功能

本次后端没有删除现有汇总分析接口，也没有下线原有风险源清单、数据安全风险清单、风险处置建议列表与保存能力。

## 二、已修改的功能

### 1. 数据安全风险清单列表

接口保持不变：

```http
GET /api/v1/projects/{projectId}/risk-items?pageNo=1&pageSize=10
```

单条风险记录新增以下字段，前端可用于展示危害程度辅助判定结果和状态：

| 字段 | 说明 |
| --- | --- |
| `harmDescription` | 风险危害程度描述 |
| `harmImpactObject` | 影响对象 |
| `harmExample` | 风险危害程度示例 |
| `harmAnalysisTrace` | 四步判定过程 |
| `harmAnalysisConfidence` | 大模型或规则判定置信度 |
| `harmAnalysisInputHash` | 判定输入摘要，用于确认应用前校验 |
| `harmAnalysisStatus` | `PENDING`、`CONFIRMED`、`MANUAL` |

说明：

- `harmLevel` 仍是正式风险清单的危害程度字段。
- 调用辅助判定建议接口不会修改 `harmLevel`。
- 用户确认后调用应用接口，后端才写入 `harmLevel`、危害程度描述、影响对象、示例和判定过程。
- `riskLevel` 由 `harmLevel` 和 `possibilityLevel` 联动计算，计算矩阵使用当前项目 `riskMatrixId` 选择的知识库风险评价矩阵。
- 前端在保存修改或确认应用后，应使用接口返回的 `riskLevel` 刷新表格，不要继续展示旧值，也不要按固定默认矩阵写死计算。

默认风险等级联动矩阵示例：

| 发生可能性 \ 危害程度 | `VERY_HIGH` 很高 | `HIGH` 高 | `RELATIVELY_HIGH` 较高 | `MEDIUM` 中 | `LOW` 低 |
| --- | --- | --- | --- | --- | --- |
| `HIGH` 高 | `MAJOR` 重大安全风险 | `MAJOR` 重大安全风险 | `MEDIUM` 中安全风险 | `LOW` 低安全风险 | `SLIGHT` 轻微安全风险 |
| `MEDIUM` 中 | `MAJOR` 重大安全风险 | `HIGH` 高安全风险 | `MEDIUM` 中安全风险 | `LOW` 低安全风险 | `SLIGHT` 轻微安全风险 |
| `LOW` 低 | `MEDIUM` 中安全风险 | `MEDIUM` 中安全风险 | `LOW` 低安全风险 | `SLIGHT` 轻微安全风险 | `SLIGHT` 轻微安全风险 |

说明：上表只表示默认矩阵。如果项目在创建/编辑时选择了其他风险评价矩阵，后端会按项目选择的矩阵计算 `riskLevel`。

### 2. 风险危害程度写入方式

原有手工保存接口仍保留：

```http
PUT /api/v1/projects/{projectId}/risk-items/{riskItemId}
```

前端如果使用大模型辅助判定流程，不建议直接调用该接口写入 `harmLevel`，应改为：

1. 调用建议接口获取候选判定。
2. 展示 `harmLevel`、`harmDescription`、`harmImpactObject`、`harmExample`、`harmAnalysisTrace` 给用户确认。
3. 用户确认后调用应用接口写入正式风险清单。

## 三、新增的功能

### 1. 单行风险危害程度建议

```http
POST /api/v1/projects/{projectId}/risk-items/{riskItemId}/harm-analysis/suggest
```

请求体通常传空对象即可：

```json
{}
```

后端会读取该风险行的 `riskTypes`、`riskDescription`、`riskSourceDescription` 和项目 `systemType`，优先调用 DashScope 大模型进行辅助判断，再由后端规则闭环匹配数据安全保护等级和风险危害程度等级。

为提升判定结果多样性和贴近人工判断，本接口还会把该风险行对应的现场测评上下文作为输入传给大模型：

| 输入字段 | 来源 | 说明 |
| --- | --- | --- |
| `assessmentCategory` | 风险行对应现场测评项 `category` | 现场测评一级分类 |
| `assessmentSubcategory` | 风险行对应现场测评项 `subcategory` | 现场测评二级分类 |
| `checkPoint` | 风险行对应现场测评项 `checkPoint` | 检查要点 |
| `evaluationResult` | 现场测评记录 `evaluationResult` | 符合情况，如 `PARTIAL`、`NON_COMPLIANT` |
| `evaluationRecord` | 现场测评记录 `evaluationRecord` | 评估结果/现场记录 |

说明：前端不需要在建议接口请求体中传这些字段，后端会根据 `riskItemId` 自动读取。

返回示例：

```json
{
  "riskRecordId": "riskrec-001",
  "harmLevel": "RELATIVELY_HIGH",
  "harmDescription": "一旦发生数据安全风险，对国家安全和经济运行产生有限危害，对社会秩序和公共利益产生危害。",
  "harmImpactObject": "社会秩序和公共利益",

    "step2Basis": [
      "项目系统类别：管理信息系统",
```

`llmStatus` 取值说明：

| 值 | 说明 |
| --- | --- |
| `SUCCESS` | 已调用外部大模型并获得结构化结果 |
| `FALLBACK` | 外部大模型调用失败，后端使用规则兜底 |
| `DISABLED` | 未启用大模型或未配置 API Key，后端使用规则兜底 |
| `PROVIDED` | 请求中显式传入了 `llmResult`，用于联调或回放 |

### 1.1 四步判定过程前端展示规范

风险危害程度辅助判定弹窗不要直接遍历整个建议对象，也不要直接渲染 `Object.values(suggestion)` 或 `Object.values(harmAnalysisTrace)`。否则会把 `[]`、`SERIOUS`、`2`、`qwen-plus`、`dashscope`、`SUCCESS` 等机器字段展示出来，用户无法理解。

弹窗建议分成“判定结果摘要”和“四步判定过程”两块展示。

判定结果摘要建议展示：

| 展示文案 | 接口字段 | 示例 |
| --- | --- | --- |
| 风险危害程度 | `harmLevel` | `MEDIUM`，前端可映射为“中” |
| 危害程度描述 | `harmDescription` | 一旦发生数据安全风险... |
| 影响对象 | `harmImpactObject` | 其他公民、法人和组织的合法权益 |
| 危害程度示例 | `harmExample` | 对电力企业造成一定的经济损失... |
| 模型状态 | `llmStatus` | `SUCCESS` / `FALLBACK` |
| 模型名称 | `llmModel` | `qwen-plus` |

四步判定过程只展示以下 4 个字段：

| 步骤 | 前端标题 | 接口字段 | 说明 |
| --- | --- | --- | --- |
| 1 | 明确被评估对象类别 | `harmAnalysisTrace.step1` | 说明项目系统类别是什么，以及为什么按该类别约束可侵害客体和侵害程度。 |
| 2 | 确定受侵害客体及侵害程度 | `harmAnalysisTrace.step2` | 说明优先判断出的侵害对象和侵害程度。 |
| 3 | 匹配数据安全保护等级 | `harmAnalysisTrace.step3` | 说明根据侵害对象和侵害程度匹配到第几级数据安全保护等级。 |
| 4 | 匹配风险危害程度等级 | `harmAnalysisTrace.step4` | 说明保护等级最终映射到哪个风险危害程度等级。 |

第二步需要额外展示“为什么这样判断”，请在 `step2` 下方增加说明区：

| 展示文案 | 接口字段 | 展示方式 |
| --- | --- | --- |
| 模型判断原因 | `harmAnalysisTrace.step2Reason` | 普通文本，展示大模型或规则兜底给出的判断理由 |
| 判定使用的信息 | `harmAnalysisTrace.step2Basis` | 列表展示，说明模型通过哪些输入字段和规则判断 |
| 模型引用证据 | `harmAnalysisTrace.step2Evidence` | 可折叠列表展示；为空时不展示 |

第二步的判定逻辑说明：

1. 后端把项目系统类别、风险类型、风险描述、风险源描述、现场测评分类、检查要点、符合情况、评估结果/现场记录作为输入传给模型。
2. 模型先按“国家安全 > 社会秩序和公共利益 > 公民、法人和其他组织合法权益”的优先级判断受侵害客体。
3. 模型再结合数据泄露、丢失、篡改、中断等风险可能造成的影响范围和严重性判断侵害程度。
4. 后端会用危害模型规则校验模型输出；如果输出和规则冲突，会返回 `needsManualReview=true`。

以下字段不建议作为“四步判定过程”逐条展示，只可作为隐藏调试信息或折叠详情：

| 字段 | 用途 |
| --- | --- |
| `systemCategory` | 机器枚举，系统类别编码 |
| `impactedObject` | 机器枚举，受侵害客体编码 |
| `damageDegree` | 机器枚举，侵害程度编码 |
| `dataSecurityProtectionLevel` | 中间计算值 |
| `reason` | 大模型判断理由，可放在“模型判断理由”单独区域 |
| `evidence` | 大模型引用证据，可放在“依据”折叠区 |
| `conflicts` | 规则冲突信息，仅 `needsManualReview=true` 时展示 |
| `ruleId` | 后端规则 ID，普通用户不需要看到 |
| `llmProvider`、`llmModel`、`llmError` | 模型调用元数据，建议放到状态或错误提示区域 |

前端伪代码：

```ts
const suggestion = response.data
const trace = suggestion.harmAnalysisTrace || {}

const summaryItems = [
  { label: '风险危害程度', value: harmLevelNameMap[suggestion.harmLevel] || suggestion.harmLevel },
  { label: '危害程度描述', value: suggestion.harmDescription },
  { label: '影响对象', value: suggestion.harmImpactObject },
  { label: '危害程度示例', value: suggestion.harmExample },
]

const stepItems = [
  { title: '第一步：明确被评估对象类别', content: trace.step1 },
  {
    title: '第二步：确定受侵害客体及侵害程度',
    content: trace.step2,
    reasonTitle: '模型判断原因',
    reason: trace.step2Reason,
    basisTitle: '判定使用的信息',
    basis: trace.step2Basis || [],
    evidenceTitle: '模型引用证据',
    evidence: trace.step2Evidence || []
  },
  { title: '第三步：匹配数据安全保护等级', content: trace.step3 },
  { title: '第四步：匹配风险危害程度等级', content: trace.step4 },
].filter(item => item.content)
```

### 2. 批量风险危害程度建议

```http
POST /api/v1/projects/{projectId}/risk-items/harm-analysis/suggest-batch
```

请求示例：

```json
{
  "riskItemIds": ["riskrec-001", "riskrec-002"],
  "limit": 20
}
```

说明：

- `riskItemIds` 为空时，默认分析当前项目全部当前风险行。
- 该接口只返回建议，不写入正式风险清单。
- 前端建议先控制批量数量，避免一次性触发过多大模型调用。

### 3. 应用风险危害程度建议

```http
POST /api/v1/projects/{projectId}/risk-items/{riskItemId}/harm-analysis/apply
```

请求示例：

```json
{
  "suggestion": {
    "riskRecordId": "riskrec-001",
    "harmLevel": "RELATIVELY_HIGH",
    "harmDescription": "...",
    "harmImpactObject": "社会秩序和公共利益",
    "harmExample": "...",
    "harmAnalysisTrace": {},
    "harmAnalysisInputHash": "..."
  }
}
```

说明：

- 仅当用户确认建议无误后调用。
- 后端会校验 `harmAnalysisInputHash`，如果风险描述等输入已变化，会返回 `HARM_ANALYSIS_STALE`。
- 写入成功后，正式风险清单行的 `harmLevel` 即为确认后的危害程度。
- 如果该行已有 `possibilityLevel`，后端会按项目风险评价矩阵自动重算 `riskLevel`。

## 四、后端大模型配置

后端读取以下环境变量：

```env
LLM_ENABLED=true
DASHSCOPE_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
DASHSCOPE_MODEL=qwen-plus
DASHSCOPE_API_KEY=replace-with-your-dashscope-api-key
DASHSCOPE_TIMEOUT_MS=30000
```

注意：

- `DASHSCOPE_API_KEY` 不会通过接口返回，前端不需要也不应该保存该 Key。
- 大模型不可用时，建议接口仍可返回规则兜底结果，页面可通过 `llmStatus` 提示用户。
