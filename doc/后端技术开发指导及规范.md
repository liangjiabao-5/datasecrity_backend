# 后端技术开发指导及规范

## 1. 后端建设目标

后端负责承载项目全流程数据、模型计算、导入导出、文件管理、报告生成、权限审计和大模型辅助能力。系统设计要保证评估项目数据可追溯、模型计算可复现、报告生成有依据、知识库变更不影响历史项目。

## 2. 推荐技术栈

推荐：

- Python 3.11+
- Flask 3.x
- Flask-SQLAlchemy 或 SQLAlchemy 2.x
- Alembic 或 Flask-Migrate，用于数据库迁移
- Flask-JWT-Extended 或自研 JWT 中间件，用于认证鉴权
- Pydantic v2 或 Marshmallow，用于请求/响应 Schema 校验
- MySQL 8 或 PostgreSQL 15
- Redis
- Celery 或 RQ，用于报告生成、导入导出、大模型调用等异步任务
- MinIO 或兼容 S3 对象存储
- openpyxl，用于 Excel 导入导出
- python-docx、docxtpl 或 docxcompose，用于 Word 报告模板填充
- apispec/flask-smorest 或 flasgger，用于 OpenAPI/Swagger 接口说明
- pytest，用于单元测试和接口测试

## 3. 后端分层结构

建议采用 Flask 单体分层架构，按领域模块拆分 Blueprint、Service、Repository、Schema 和 Model。目录建议如下：

```text
app/
  __init__.py               Flask app factory
  config.py                 环境配置
  extensions.py             db、redis、jwt、migrate、celery 等扩展
  blueprints/
    auth                    认证、授权、用户、角色
    project                 项目管理、项目配置、项目状态
    basic                   项目基本信息、评估依据、被评估单位
    plan                    评估团队、被评估方团队、关注点、差距项
    survey                  信息调研、系统、数据资产、处理活动、防护措施
    evaluation              现场测评、检查项、评估记录、评分计算
    risk                    风险源、风险清单、处置建议、风险等级计算
    knowledge               知识库、模板、模型、矩阵、建议、案例
    report                  报告任务、报告模板、报告文件
    file                    文件上传、下载、对象存储
    llm                     大模型辅助、提示词、审计
  services/                 领域业务服务
  repositories/             数据访问封装
  models/                   SQLAlchemy ORM 模型
  schemas/                  Pydantic/Marshmallow Schema
  tasks/                    Celery/RQ 异步任务
  common/                   通用响应、异常、分页、字典、工具
  audit/                    操作日志、业务审计
```

## 4. 核心领域模型

### 4.1 项目域

| 对象 | 说明 |
| --- | --- |
| Project | 项目主记录，包含项目编号、名称、评估单位、状态、负责人、描述 |
| ProjectConfigSnapshot | 项目配置快照，保存评估模板、系统类型、评分模型、风险危害模型、风险评价矩阵 |
| ProjectTemplate | 项目模板，由项目配置沉淀形成 |
| ProjectMember | 项目团队成员，支持外部人员 |

设计要求：

- 项目编号允许重复，项目唯一识别使用项目 ID；前后端不得用项目编号作为路由、缓存、选择或关联的唯一键。
- 项目配置引用知识库时要保存名称、版本和快照内容。
- 项目删除采用逻辑删除。

### 4.2 知识库域

| 对象 | 说明 |
| --- | --- |
| AssessmentTemplate | 评估项模板 |
| AssessmentItem | 检查项 |
| RemediationSuggestion | 风险处置建议 |
| ScoreModel | 评分模型 |
| HarmModel | 风险危害程度分析模型 |
| RiskMatrix | 风险评价矩阵 |

设计要求：

- 模板、模型、矩阵必须版本化。
- 被项目引用的条目不可物理删除，只允许停用。
- 评分模型保存公式、结果分值、可能性等级区间。
- 风险矩阵保存可计算二维结构。

### 4.3 调研域

| 对象 | 说明 |
| --- | --- |
| BusinessSystem | 业务和信息系统 |
| DataAsset | 数据资产 |
| PersonalInfoAsset | 个人信息情况 |
| ImportantDataAsset | 重要数据 |
| CoreDataAsset | 核心数据 |
| DataProcessingActivity | 数据处理活动 |
| ProcessingActivitySurvey | 数据处理活动整体问卷 |
| SecurityProtectionSurvey | 安全防护措施调研 |

设计要求：

- 新版数据处理活动优先按整体问卷保存，旧版行记录接口保留兼容，用 `activityType` 区分七类处理活动。
- 数据资产可关联业务系统。
- 文件上传只保存文件 ID，不保存本地路径。

### 4.4 测评域

| 对象 | 说明 |
| --- | --- |
| ProjectAssessmentItem | 项目检查项快照 |
| EvaluationRecord | 现场测评记录 |
| ScoreCalculationRecord | 分数计算记录 |

设计要求：

- 项目创建或开始评估时，从评估模板生成项目检查项快照。
- 评估记录保存符合情况（`evaluationResult`）和评估结果（`evaluationRecord`）。
- 现场测评记录不再保存问题描述、整改建议和风险类型。
- 分数计算记录保存评分模型版本和计算明细。

### 4.5 风险域

| 对象 | 说明 |
| --- | --- |
| ProjectRiskSummaryRecord | 风险汇总记录，统一承载风险源、风险清单和处置建议三个页面投影 |
| RiskSourceTemplate | 国标风险源模板，用于刷新汇总分析时匹配和填充风险字段 |

设计要求：

- 一条基本符合或不符合现场测评记录对应一条风险汇总记录。
- 刷新时按工作表、评估类别、评估子类、评估项匹配国标风险源模板，填充评估类别、评估子类、风险描述、整改建议、风险源描述、风险源类型和风险类型。
- 风险记录要能追溯来源检查项、检查要点、评估结果、数据资产和处理活动。
- 自动生成内容和人工确认内容分开保存，报告读取最终确认内容。
- 刷新统计数据不得静默覆盖人工修改。
- 旧 `risk_source`、`risk_item`、`risk_suggestion` 表下线后不再作为持久化模型使用。

### 4.6 报告域

| 对象 | 说明 |
| --- | --- |
| ReportTask | 报告生成任务 |
| ReportRecord | 报告记录 |
| ReportTemplate | 报告模板 |
| ReportSectionSelection | 报告章节选择 |

设计要求：

- 报告生成走异步任务。
- 报告记录保存类型、状态、生成时间、创建人、文件 ID、失败原因。
- 报告文件下载必须做项目权限校验。

## 5. 状态设计

### 5.1 项目状态

| 状态 | 含义 | 触发 |
| --- | --- | --- |
| `NOT_STARTED` | 待开始 | 项目创建完成 |
| `IN_PROGRESS` | 进行中 | 点击开始评估或进入项目流程 |
| `COMPLETED` | 已完成 | 报告交付确认 |
| `ARCHIVED` | 已归档 | 项目关闭或归档 |

### 5.2 报告状态

| 状态 | 含义 |
| --- | --- |
| `PENDING` | 已创建任务，等待执行 |
| `RUNNING` | 正在生成 |
| `SUCCESS` | 生成完毕 |
| `FAILED` | 生成失败 |
| `DELETED` | 已删除或不可下载 |

### 5.3 知识库状态

| 状态 | 含义 |
| --- | --- |
| `ENABLED` | 启用 |
| `DISABLED` | 停用 |
| `DRAFT` | 草稿 |

## 6. 计算规则

### 6.1 评分模型

默认公式：

```text
可能性计算分值 =
（符合项数量 × COMPLIANT分值
 + 基本符合项数量 × PARTIAL分值
 + 不符合项数量 × NON_COMPLIANT分值）
÷（符合项数量 + 基本符合项数量 + 不符合项数量）
× 100%
```

要求：

- 不适用项不进入分子和分母。
- 评分模型可配置符合、基本符合、不符合、不适用的基础分值。
- 可能性区间必须覆盖 0%-100%，无重叠、无空档。
- 区间命中按 `min <= score < max` 执行；为覆盖满分，`score = 100` 且区间 `max = 100` 时命中该区间。
- 计算结果保存得分、可能性等级、模型版本和计算明细。

### 6.2 风险等级

风险等级由三部分决定：

- 风险发生可能性：来自现场测评评分模型。
- 风险危害程度：来自风险危害程度分析模型，也可人工复核。
- 风险评价矩阵：根据可能性和危害程度映射风险等级。

输出风险等级建议枚举：

- `MAJOR`：重大安全风险
- `HIGH`：高安全风险
- `MEDIUM`：中安全风险
- `LOW`：低安全风险
- `SLIGHT`：轻微安全风险

## 7. 导入导出

### 7.1 导入

导入接口统一采用任务化或半任务化处理：

- 小文件可同步解析，但仍要返回错误明细。
- 大文件建议创建导入任务，前端轮询状态。
- 错误明细至少包含行号、字段、错误原因。

导入校验：

- 必填字段。
- 枚举值。
- 项目 ID 是否存在。
- 检查项编号是否存在。
- 日期和联系方式格式。
- 文件类型和大小。

### 7.2 导出

导出可同步返回文件流，也可生成导出任务。以下场景建议任务化：

- 现场测评全量导出。
- 信息调研全量导出。
- 报告生成。

## 8. 文件管理

文件类型：

- 拓扑图、数据流转图。
- 导入模板。
- 导入错误明细。
- 导出文件。
- 报告文件。

要求：

- 文件上传返回 `fileId`。
- 业务表只保存 `fileId`。
- 下载接口校验用户是否有项目权限。
- 文件对象存储路径不要暴露给前端。
- 文件删除采用引用检查，已被报告或项目引用的文件不直接删除。

## 9. 权限与审计

### 9.1 角色建议

| 角色 | 权限范围 |
| --- | --- |
| 管理员 | 知识库、模板、模型、矩阵、用户权限 |
| 项目负责人 | 项目创建、配置、团队、流程推进、报告生成 |
| 评估人员 | 信息调研、现场测评、风险分析、整改建议 |
| 查看员 | 查看项目、清单、报告 |

### 9.2 审计记录

以下操作必须记录：

- 新增、编辑、删除项目。
- 开始评估、项目状态变更。
- 导入、导出。
- 批量修改评估结果。
- 分数计算、风险等级重新计算。
- 刷新风险统计数据。
- 生成报告、删除报告、下载报告。
- 知识库模板、模型、矩阵变更。
- 大模型调用和应用建议。

审计字段：

- 操作人、操作时间、操作类型、业务对象、对象 ID、变更前摘要、变更后摘要、请求来源 IP。

## 10. 报告生成服务

建议流程：

1. 前端提交报告生成请求，包括项目 ID、报告名称、报告类型、章节选择。
2. 后端创建 `ReportTask`，状态为 `PENDING`。
3. 异步任务读取项目最终确认数据。
4. 按报告模板填充章节、表格、风险清单、处置建议。
5. 统一生成表格编号、图示编号和引用编号。
6. 文件上传到对象存储。
7. 更新报告记录为 `SUCCESS`，失败则记录 `FAILED` 和错误摘要。

要求：

- 报告任务可重试。
- 同一项目允许多份报告。
- 生成期间不锁定项目，但报告使用任务开始时的数据快照。

## 11. 大模型服务边界

大模型仅作为辅助能力：

- 可以生成评估结果建议。
- 可以生成整改建议。
- 可以做案例匹配解释。
- 可以润色报告片段。

不允许：

- 直接替代评分模型计算。
- 直接替代风险等级判定。
- 未经用户确认写入最终报告。
- 默认发送个人手机号、邮箱、联系人等敏感信息。

## 12. 异常处理规范

统一异常类型建议使用 Python 自定义异常类，并在 Flask 全局错误处理中转换为统一响应：

- `BusinessError`：业务校验失败。
- `PermissionError` 或 `AccessDeniedError`：无权限。
- `NotFoundError`：资源不存在。
- `DuplicateError`：编号或名称重复。
- `ImportError` 或 `ImportTaskError`：导入失败。
- `ReportGenerationError`：报告生成失败。
- `LlmError`：大模型调用失败。

接口错误返回应包含：

```json
{
  "code": "VALIDATION_ERROR",
  "message": "请求参数不合法",
  "data": null,
  "traceId": "..."
}
```

## 13. 开发规范

- Blueprint/View 函数只处理路由、参数、权限和响应，不写业务逻辑。
- Service 负责业务流程、事务边界和跨表一致性。
- Repository 负责数据访问，复杂查询统一封装，不在 View 函数中直接拼查询。
- Schema 与 ORM Model 分离，入参校验、出参序列化不得直接暴露内部模型。
- 对外接口字段使用驼峰命名。
- 所有列表接口必须分页。
- 所有删除接口默认逻辑删除。
- 关键业务操作必须在 SQLAlchemy session 事务中执行，失败时回滚。
- 模型、矩阵、模板变更必须生成新版本。
- Flask 应采用 app factory 模式，配置按开发、测试、生产环境拆分。
- 异步任务放入 `tasks/`，任务入参只传业务 ID，不传完整大对象。

## 14. 测试要求

后端至少覆盖：

- 项目创建、编辑、状态切换。
- 项目编号可重复场景，确保创建、编辑、查询和后续业务操作均以项目 ID 定位。
- 评分模型区间合法性。
- 分数计算。
- 风险等级矩阵计算。
- 刷新风险源不覆盖人工确认数据。
- 报告任务成功和失败。
- 文件上传下载权限。
- 知识库条目被引用后不可物理删除。
