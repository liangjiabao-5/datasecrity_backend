# 汇总分析分页与发生可能性填充功能修改说明

更新时间：2026-06-02

## 1. 删除或废止内容

- 本次前端未删除页面入口、接口地址或风险汇总字段。
- 废止原“刷新统计数据时 `possibilityLevel` 默认固定为 `"-"`”的处理逻辑。刷新后应按现场测评分数和项目评分模型自动填充发生可能性。
- 废止风险源清单、风险清单、处置建议列表仅返回全量数组的实现方式。前端已传入分页参数，后端应按分页返回。

## 2. 修改内容

### 2.1 汇总分析三张表增加分页

涉及页面：

- 数据安全风险源清单
- 数据安全风险清单
- 数据安全风险处置建议

涉及接口：

```http
GET /api/v1/projects/{projectId}/risk-sources?pageNo=1&pageSize=10
GET /api/v1/projects/{projectId}/risk-items?pageNo=1&pageSize=10
GET /api/v1/projects/{projectId}/risk-suggestions?pageNo=1&pageSize=10
```

建议统一返回：

```json
{
  "list": [],
  "pageNo": 1,
  "pageSize": 10,
  "total": 0
}
```

前端当前兼容 `list`、`records`、`rows`、`content` 和裸数组，但后端目标格式建议使用 `list`。

### 2.2 表格增加自增序号

三张表均已增加“序号”列。序号由前端按分页计算：

```text
序号 = (pageNo - 1) * pageSize + 当前页行下标 + 1
```

后端无需新增序号字段。

### 2.3 刷新统计数据时填充发生可能性

涉及接口：

```http
POST /api/v1/projects/{projectId}/risk-summary/refresh
```

刷新时后端需要：

1. 读取项目当前绑定的评分模型。
2. 使用现场测评页面“分数计算”相同口径计算当前系统得分。
3. 使用该评分模型的 `possibilityRanges` 判断得分所在区间。
4. 将区间对应的 `level` 写入风险汇总记录的 `possibilityLevel`。

分数计算口径：

```text
score = (符合项数量 * COMPLIANT分值
       + 基本符合项数量 * PARTIAL分值
       + 不符合项数量 * NON_COMPLIANT分值)
       / (符合项数量 + 基本符合项数量 + 不符合项数量)
       * 100
```

`NOT_APPLICABLE` 不计入分子和分母。

区间判断规则：

```text
min <= score < max
```

即左闭右开，包含左边界，不包含右边界。示例：配置 `[30, 75)` 时，`score = 30` 命中该区间，`score = 75` 不命中该区间。

后端规则补充：为覆盖满分，`score = 100` 且区间 `max = 100` 时命中该区间。

### 2.4 人工修改覆盖规则

刷新请求仍使用现有请求体：

```json
{
  "overwriteManualChanges": false
}
```

建议后端规则：

- `overwriteManualChanges = false`：已人工修改的风险汇总记录保持原值；未人工修改记录自动刷新 `possibilityLevel`。
- `overwriteManualChanges = true`：允许按最新现场测评结果和评分模型覆盖刷新。

## 3. 需要后端同步确认

- 风险汇总列表接口支持 `pageNo`、`pageSize` 入参，并返回 `total`。
- 现场测评“分数计算”接口和汇总分析“刷新统计数据”使用同一套评分模型区间映射，不再写死高/中/低阈值。
- 评分模型区间按左闭右开执行；后端已明确 `score = 100` 且区间 `max = 100` 时命中该区间。
